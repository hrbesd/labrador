# Reactor模块技术说明

*不完整*

*可用来指导实现*

## 目的

实现需求文档中```网页改造```部分。

## 接口

工作流程：

- 前：Parser
	* 目前在同一个主机运行，通过文件系统共享XML数据库
	* 今后计划拓展出独立的网络接口（FTP或Web Service）
- 后：Assembler
	* 目前与Assembler原有的接口形式不变
	
## 改造

从以下三方面入手改造:

* 无障碍化
* 标准化
* 结构化和语义化

### 无障碍化

*注：相当于目前的 `HarbinGovParser::cleanBodyData`*

* 检测到下列标签，直接删除标签： __A__
	- `<!--...-->` (内置脚本的另行判断)
	
* 检测到下列标签，在日志中留下警告，并可根据设置选择性删除： __A__

	- 缺失`alt`属性`<img>`
	- `<applet>`
	- `<area>`
	- `<map>`
	- `<iframe>`
	- `<script>`
	- `<style>`
	- `<object>`
	- `<audio>`和`<video>`
	- `type`属性是`audio`或`video`的`<embed>`
	- 带有`action`属性的`<form>`
	- 内容空白的`<canvas>`

* 检测到下列标签，在日志中留下警告： __AA__

	- 缺失`summary`的`<table>`
	- 太小的`<textarea>` (`rows` ≤ 3, `cols` ≤ 20)
	- `<figure>`
	- `<bb>`

* 根据元素的Id查找人工数据库: __AA__
	- 增加或替换`<img>`的`alt`内容
	- 在`<object>`或`<embed>`标签的附近（视觉范畴）插入补充文本
	- 为`<video>`和`<audio>`标签添加`<track>`
	
* 对于多媒体内容(`<embed>` `<object>` `<audio>` `<video>`)，自动通过软件转码，提供更多格式 __AAA__

- 链接改造
	* 如果`<a>`的`target`属性是`_blank`，改为`_self` __A__
	* 如果为站内链接，为`<a>`的`title`属性追加目标网页的标题 __AAA__
	* 如果为站内链接，为`<a>`的`title`属性追加"点击此链接将转出本网站"字样 __AA__
	* 为`<a>`标签增加`rev`和`rel`属性 __AAA__
	
* 根据网页的Id，应用代码级的替换规则 __AAA__
	
### 标准化

- 检测到不封闭的HTML时，在日志中留下警告 __AA__
- 封闭下列标签: __AA__
	* `<base>`
	* `<br>`
	* `<hr>`
	* `<img>`
	* `<option>`
	* `<embed>`
- 删除下列标签: __A__	
	* `<font>`
	* `<big>`
	* `<basefont>`
	* `<link>`
- 替换下列标签: __AA__	
	- `<big>` ~ `<span style="font-size:120%">`
	- `<center>` ~ `<p style="text-align:center">`
	- `<dir>` ~ `<ul>`
	- `<menu>`改成`<ul>`
	- `<div>`(如果内容空白) ~ `<br />`
	- `<tt>` ~ `<pre>`
	- `<marquee>` ~ `<p>`
	
- 检测长度超过20个字符的编码或某种语言，尝试设计通用算法插入`<wbr>`
- 检测长度超过20个字符的编码或某种语言，用`<data>`包裹，`value`属性就是自身

### 结构化和语义化

- 替换下列标签: __A__
	- `<i>` ~ `<em>`
	- `<b>` ~ `<strong>`
	- `<s>` ~ `<del>`
	- `<u>` ~ `<ins>`

- 尝试识别小标题并转成`<h2>..<h6>` __AAA__
- 尝试根据标题级为正文划分区域，包裹`<section>`标签 __AAA__
- 尝试识别列表，转成`<ul>`，不改变原有文本内容 __AAA__
- 将`<font>`尝试替换成`<strong> <h1> <h2> <mark>`等标签 __AAA__
- 尝试识别地址，使用`<address>`标签表示 __AAA__
- 尝试理解连续两个以上的`<br>`，并使用CSS的`margin`代替 __AAA__
- 尝试理解连续两个以上的`<p>`，并使用CSS的`margin`代替 __AAA__
- 尝试理解连续五个以上的空格，并使用CSS的`align`代替 __AAA__
- 尝试在上下文寻找**来源**之类的字眼，为`<blockquote>`和`q`补充`cite`属性 __AAA__
- 尝试把复杂的`<table>`重新用`<thead> <tbody> <tfoot> <thead>`构建 __AAA__
- 为正文添加`<article>`标签 __AA__

- 为时间和日期添加`<time>`标签 __AA__

- 为文章开头结尾版权、隐私、来源等声明包裹`<footer>`标签 __AA__

### 替换站内连接

- 如果遇到绝对URL，就利用parser提供的全站xml查找对应的hash值，如果没有找到，说明是外链
- 对于外链，添加跳转警告
- 如果遇到相对URL，利用文件中页面的URL和相对URL，计算生成绝对URL，执行查找
- 如果找到就利用hash替换，否则保留输出log

## 断句

原始文本句子前后插入`<span id="<hash>" class="yz_sentence">..</span>`，根据句号、问号、感叹号、分号断句。

可选方案：使用HTML5 `<data>`标签。`value`属性就是文本的hash。

要求:

- 句子上限100字，否则在逗号断或者在第100字断 __A__
- 考虑Unicode，标点集合中包括所有语言的句号类型的标点 __AAA__
- 兼容中英文混合断句 __AA__

## 模块

### 日志

- 格式: [Time] [Type] [Column] [Message]
- 类型如下：
	- 信息 [Info]
	- 警告 [Warning]
	- 错误 [Error]
	- 调试 [Debug]
	
- 在同一栏目下，不重复纪录类似的事件 __AAA__

### 规则

#### 规则定义

规则采用自己定义的DSL语法来进行编写，基本的语法规则如下：

- 用`#`代表注释
- 规则是由许多“块”组成的
- 每一个块由开头、条件、操作、输出（日志）和结束四个部分组成
- 开头：每一块以一个元素名称或者元素名称下面对应的某个属性开头（元素名称与属性之间使用空格分割）
- 条件：条件使用`->`符号进行表示；每一块可以有多个`->`条件，条件中的`+`表示包含，`-`表示缺失；在条件中还可以使用表达式语法，比如等于、大于等于、小于等于等等
- 操作：操作使用`=>`符号表示，如果开头与条件都得到了匹配，那么就执行对应的操作。可能的操作类型包括：
  * `remove`：删除这个标签
  * `fill(dataSourcePath)`：从外部数据源获取数据，来填充标签所缺失的内容或属性
  * `replace(target)`：用target标签替换原来的标签
  * `set(value)`：设置某个属性的值
  * `append(value)`：为某个属性追加部分信息
- 输出：输出使用`==>`符号表示，在处理过程中可能遇到无法自动处理的内容，需要记录日志
  * 由两部分组成：`level | message`，分别对应日志等级和日志记录的消息
  * 日志的level包括：`info`,`warning`,`error`,`debug`
- 结束：为了区分段落之间的内容，使用`===`作为某一个段落的结束，段落终结之后，开始下一个段落
- 其他：
  * 由于有些标签要判断内容是否为空，这个无法表示，所以定义了一个关键字`contents`，表示标签的内容
  * 由于有些规则需要判断标签中某个属性信息是否为空，所以定义了一个关键字`values`，表示标签某个属性的内容
  * 为了方便一些处理，需要提供一些标准的库函数，包括：
  * `insideSite(pageURL)`：根据url信息判断这个页面是否已经经过处理
  * `outsideSite(pageURL)`：根据url信息判断这个页面是否没有经过处理
  * 如果多个块匹配的是同一个元素（块开头一致），那么在文件中定义靠前的块规则首先执行

#### 规则存储

- 以网站为单位独立存储
- 规则存储文件的后缀是`.rrule`
- UTF-8编码的文本文件
- 为了适应某个栏目，文件名采用类似于包结构的方式进行命名：比如`哈尔滨市网站.政府公告.2012年政府公告.rrule`

#### 规则批处理

- 用于若干规则的批量执行
- 编辑规则的执行顺序
- UTF-8编码的文本文件，内容就是规则的文件名列表


## 文件生成

#### 数据来源

- 数据来源于parser分析之后的文件内容，是以xml形式存储的文件内容。reactor针对这个文件做处理。
- 生成的文件名称与parser提供的文件名称保持一致。都采用hash的方法进行保存。

#### 海量文件处理

- 当一个文件夹中文件过多影响效率的时候，就建立目录结构，其中目录是hash文件的前2位。所有hash文件名前2位相同的文件归入同一个目录中
- 如果后期能使用海量文件系统，那么直接使用文件系统替换
